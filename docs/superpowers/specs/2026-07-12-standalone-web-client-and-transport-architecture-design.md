# Standalone Web Client and Transport Architecture Design

**Date:** 2026-07-12
**Status:** Approved design

## Objective

Make Switchboard fully usable without Discord through a responsive, installable web client with a conversation experience comparable to the ChatGPT and Claude clients. Discord remains fully supported, but becomes one transport adapter among a stable family of adapters intended to include Slack, Teams, and custom integrations.

Switchboard owns canonical conversations. Web and external platforms are surfaces attached to those conversations rather than sources of application identity or state.

## Scope

This design covers:

- Transport-neutral conversations and message persistence.
- A responsive browser client that can be installed as a PWA.
- Continued full Discord support through an adapter boundary.
- A stable extension contract for future Slack, Teams, and custom adapters.
- Configurable two-way, inbound-only, outbound-only, and notifications-only transport links.
- Agent selection, consultation, delegation, handoff, approvals, and operational visibility in the web client.
- Incremental migration from Discord-channel-owned state.

This design does not introduce native Switchboard login, require an external database, build Slack or Teams adapters, or create an Electron/Tauri application.

## Product Decisions

- The first client is a responsive browser application with optional PWA installation.
- Authentication remains deployment-owned. Switchboard trusts a configurable identity header supplied by an existing SSO or reverse proxy.
- SQLite is the default canonical store, accessed behind repository interfaces that permit a later PostgreSQL implementation.
- Each conversation has one primary agent. Users may switch it, consult another agent, delegate background work, or explicitly hand off the conversation.
- A conversation may link to several external surfaces. New links default to two-way synchronization.
- Per-link sync modes are `two_way`, `inbound_only`, `outbound_only`, and `notifications_only`.
- The implementation evolves the existing Bun/TypeScript modular monolith rather than introducing a separate service or rewriting the platform.

## Architecture

The system is divided into five major boundaries:

1. **Web client and API:** The responsive workspace and future API clients interact with transport-neutral resources and event streams.
2. **Conversation service:** Owns conversations, participants, messages, ordering, lifecycle events, and authorization.
3. **Agent runtime:** Owns routing, primary-agent turns, consultations, delegations, handoffs, session state, and tool activity.
4. **Transport router and adapters:** Plans deliveries and invokes platform-specific adapters. Adapters normalize inbound events and render outbound events but do not orchestrate agents.
5. **Operations:** Owns approvals, audits, traces, delivery health, tools, workflows, status, and configuration.

SQLite persists canonical conversation state and delivery state. The core must start and support complete web conversations when every external transport is disabled or unavailable.

### Boundary Rules

- Canonical conversation IDs are generated by Switchboard. Discord channel IDs and other platform identifiers exist only as transport-link metadata.
- Web messages and adapter messages use the same authorization, persistence, routing, agent, audit, and live-event pipeline after normalization.
- Adapter implementations depend on the transport contract, not conversation-service internals.
- Transport failures never roll back canonical messages or make the web client unavailable.
- Existing Discord-specific rendering and event parsing move behind the Discord adapter incrementally.

## Canonical Data Model

### Conversation

Stores the canonical ID, title, primary agent, creator, creation and update timestamps, and archive state. The primary agent may change without replacing the conversation or losing history.

### Participant

Associates a user or agent identity with a conversation and a role. Initial user roles are `owner`, `member`, and `viewer`. Agent participation records distinguish the primary agent, consultants, and delegated agents.

### Message

An immutable canonical content envelope containing a conversation sequence number, author identity, origin, content parts, optional reply relationship, creation time, and processing state. Edits and deletions are lifecycle events linked to the original message rather than destructive rewrites.

### Transport Link

Connects one conversation to an adapter-specific external location. It stores adapter type, external location ID, display metadata, sync mode, enabled state, and health timestamps. A unique adapter/location constraint prevents the same external channel from mapping ambiguously to multiple active conversations.

### Delivery

Tracks delivery of a canonical event to one transport link. It stores the target link, attempt count, status, retry time, external message ID, and sanitized failure details. A unique message/link/event relationship makes delivery idempotent.

### External Event Receipt

Records adapter event identity before processing. A unique adapter/event ID constraint prevents duplicate ingestion and mirror loops.

## Message Flow

### Web-Originated Message

1. The API authenticates the proxy-supplied identity and authorizes conversation access.
2. The conversation service validates and persists the message in a SQLite transaction, assigning its sequence number.
3. A committed event is published to connected clients.
4. The primary-agent turn is queued through the existing runtime.
5. The transport router plans deliveries for eligible links according to sync mode.
6. Each delivery proceeds independently and records its outcome.

### Adapter-Originated Message

1. The adapter validates the platform event and converts it to a normalized inbound envelope.
2. The router resolves the external location to one transport link and canonical conversation.
3. The external event receipt is inserted idempotently.
4. The message enters the same canonical persistence and agent-routing pipeline used by web input.
5. The originating external event is excluded from echo delivery. Other eligible links may receive the canonical message.

### Agent Output

Agent output is persisted as canonical message or activity events before being presented. Transcript content is mirrored through eligible links. Tool use, consultation, delegation, and progress events remain structured activity unless an adapter explicitly supports an appropriate rendering or the conversation requests notifications.

## Transport Contract

Each adapter provides:

- Lifecycle methods to start, report health, reload configuration, and stop.
- Inbound event subscription producing normalized envelopes.
- Outbound send operations for text, supported rich content, attachments, edits, deletions, and platform interactions as capabilities allow.
- Stable platform event and message identifiers for deduplication.
- Identity and external-location descriptors.
- A capabilities declaration so the core can degrade unsupported features explicitly.

The core provides adapters with delivery requests and records results. Adapters do not access SQLite directly, choose agents, mutate conversation membership, or bypass audit and approval services.

Discord is migrated to this contract first and serves as its initial contract-test implementation. Slack, Teams, and custom adapters are outside the first implementation but must not require core model changes.

## Agent Participation

Each conversation has one primary agent responsible for ordinary user turns. The client supports:

- Switching the primary agent while retaining canonical history.
- A one-off consultation whose result is returned to the primary agent and represented in activity.
- Background delegation with visible progress and completion state.
- Explicit handoff that changes the primary agent and records a conversation event.

Several agents do not reply freely to the same user turn. This preserves predictable ownership and prevents response races while retaining Switchboard's multi-agent capabilities.

## Web Workspace

The client uses an expandable workspace shell with five destinations.

### Conversations

Provides conversation creation, search, archive, transcript, streaming agent output, attachments, primary-agent selection, linked surfaces, background work, and context/session status. Desktop may show an optional activity inspector beside the transcript.

### Agents

Provides agent availability, current work, capabilities, configuration, usage, session controls, and restart-impact previews using existing safe configuration flows.

### Approvals

Provides a global actionable queue. Requests also appear in their originating conversation. Web and Discord decisions call the same idempotent, audited operation.

### Operations

Provides transport health, failed deliveries and manual retry, workflows, tool activity, traces, audit ledger, runtime status, and diagnostic commands.

### Settings

Provides hub configuration, transport-adapter configuration, appearance, identity-header settings, and PWA preferences.

## Responsive and PWA Behavior

Desktop uses a collapsible application rail, conversation list, transcript, and optional inspector. Tablet collapses the inspector into a drawer. Mobile presents one primary surface at a time, with bottom navigation and full-screen drawers for secondary content.

The PWA includes a web manifest, icons, installation metadata, and a service worker that caches only the application shell and safe static assets. Canonical conversation data remains server-owned. The client may preserve an unsent local draft, but it must not imply that a message was submitted while offline. Offline and reconnecting states are prominent, and stale operational data is marked as such.

## Authentication and Authorization

Switchboard does not provide login in this phase. A configurable trusted header supplies the user identity. Deployment documentation must state that the application must not be exposed directly where callers can forge this header.

The API centralizes identity extraction and conversation authorization. Conversation roles are separate from administrative permissions. External participants retain adapter-qualified identities unless deployment-specific mapping associates them with a trusted Switchboard user.

## Live Events and Turn State

The API exposes a conversation event stream with monotonic per-conversation sequence numbers. The client resumes after reconnecting from its last received sequence and fetches any gap before applying new events.

Agent turns progress through `queued`, `working`, `streaming`, `completed`, and `failed`. User messages appear only after canonical persistence succeeds. Tool calls, tool results, consultations, delegations, and status changes appear as collapsible activity items rather than ordinary transcript messages.

The transport contract and canonical model do not depend on SSE. The initial web implementation may continue using SSE because the existing project already supports it; a future WebSocket implementation can provide the same sequenced events without changing persistence.

## Error Handling and Reliability

- SQLite transactions protect message sequence assignment, external event receipts, and delivery creation.
- Client-generated idempotency keys prevent repeated sends after request retries.
- Agent failures produce visible retryable turn states without deleting user input.
- Transport failures are isolated per delivery and retried with bounded exponential backoff and jitter.
- Exhausted deliveries remain visible in Operations and may be retried manually.
- Malformed or unverifiable adapter events are rejected or quarantined and audited without entering the conversation.
- Losing a client event-stream connection does not cancel an agent turn.
- Conflicting approval decisions return the already-resolved outcome and never execute an approved action twice.
- Adapter capability gaps produce explicit degraded behavior rather than silent data loss.

## Migration Strategy

Migration is incremental:

1. Add repository interfaces, SQLite schema, canonical conversation service, and migrations without removing existing Discord behavior.
2. Define the transport contract and place Discord behind it while preserving current gateway behavior through regression tests.
3. On first activity, map each existing Discord channel to one canonical conversation and a two-way Discord link.
4. Import reliable cached transcript entries where author, order, and origin can be preserved. Do not invent unavailable history.
5. Add transport-neutral conversation APIs and the new workspace alongside the embedded dashboard.
6. Migrate approvals, agent management, observability, and configuration into workspace destinations.
7. After feature parity and operational verification, redirect the legacy dashboard to the corresponding workspace destination and remove duplicated presentation code in a later cleanup.

The existing `channelId`-based internal APIs may remain behind compatibility adapters during migration, but new domain interfaces use `conversationId`.

## Testing Strategy

### Unit and Repository Tests

- SQLite migrations, transactions, ordering, idempotency, and repository behavior.
- Conversation authorization and agent-participation rules.
- Delivery planning for all sync modes.
- Retry scheduling, failure classification, and external identity mapping.

### Transport Contract Tests

A shared suite verifies adapter lifecycle, inbound normalization, outbound rendering, idempotency identifiers, capability declarations, error classification, and shutdown. Discord must pass this suite while all existing Discord regression tests continue to pass.

### Integration and End-to-End Tests

- Complete web-only conversation with Discord disabled.
- Web-to-Discord and Discord-to-web two-way mirroring.
- One conversation linked to multiple transports with mixed sync modes.
- Duplicate platform event and retry deduplication.
- Agent streaming, reconnect gap recovery, failure, and retry.
- Agent switching, consultation, delegation, and handoff.
- Concurrent approval decisions.
- Transport outage, retry exhaustion, Operations visibility, and manual retry.
- Trusted-header enforcement and conversation-role access.

### Client Tests

- Desktop, tablet, and mobile navigation and layout.
- Keyboard navigation, focus management, accessible labelling, and screen-reader announcements for streaming state.
- PWA installation metadata, application-shell caching, offline indicators, and preserved drafts.
- Transcript rendering, activity collapse, attachment states, and reconnection.

## Success Criteria

- Switchboard can start with Discord disabled and support durable multi-turn web conversations, agent work, approvals, and administration.
- Discord retains existing supported behavior through the adapter and passes existing regression tests.
- A message committed through any surface appears once in the canonical conversation and at most once on each eligible linked surface.
- A failed external transport cannot prevent web use or lose canonical messages.
- A new transport can be added by implementing the adapter contract and configuration without modifying the conversation data model or agent runtime.
- The client is usable across desktop, tablet, and mobile widths and satisfies browser PWA installation criteria.
- Deployment-owned SSO identity continues to work through a configurable trusted header without a Switchboard login system.

## Deferred Work

- Slack and Teams adapter implementations.
- Native Switchboard accounts or built-in SSO.
- PostgreSQL repository implementation and multi-node coordination.
- Native desktop packaging.
- Full offline message submission and synchronization.
- Cross-platform edit/delete mirroring where platform semantics cannot be represented consistently.
